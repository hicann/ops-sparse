# 错误处理指南

## 核心原则

1. **问题回退不跳过** - 验证失败时，修复后从当前阶段重新验证
2. **先定位后修复** - 不要盲目修改代码，先定位根因
3. **分层排查** - 按 Host（描述符/参数校验）→ Kernel（Tiling/计算）→ 编译环境 顺序排查
4. **记录问题** - 复杂问题创建 issue 文件，LOG.md 只放链接

## 重试阈值

| 验证类型 | 重试次数 | 说明 |
|---------|---------|------|
| 编译失败 | 3 次 | 语法或环境问题 |
| ST 精度失败 | 3 轮 | 每轮修复后重新验证 |
| 性能不达标 | 5 轮 | 优化迭代需要多次尝试 |
| 方案评审不通过 | 3 轮 | 打回设计修订后重评 |
| README 质量检查不通过 | 2 次 | 打回 writer 修复 README 后重检 |

## 阶段二：开发

### 2.1/2.2 算子开发

| 错误类型 | 处理方式 |
|---------|---------|
| 编译错误 | 检查头文件依赖、AscendC API 签名 |
| 描述符相关错误 | 检查描述符字段赋值完整性、类型匹配 |
| Tiling 错误 | 检查 TilingData 布局、UB 预算 |

### 2.x.1.B 测试开发

| 错误类型 | 处理方式 |
|---------|---------|
| 测试框架编译失败 | 检查 GTest 链接、test/frame/ 头文件依赖、CMake 宏使用是否正确 |
| CSV 格式错误 | 检查列名是否与 Param 结构体字段匹配、基础列是否完整 |
| golden 实现错误 | 对比 Eigen golden 与自写 golden 输出，定位计算逻辑差异 |
| 填充函数缺失 | 检查 test/frame/fill.h 是否有对应函数，禁止在测试文件中定义临时填充函数 |

## 阶段三：验收

### 3.1 代码检视

直接打回修复，不询问用户。连续 3 次不通过 → 生成 CP 问卷。修复后必须重新执行 3.1 代码检视验证，确保所有问题已解决。

### 3.2 性能验收

| 错误类型 | 处理方式 |
|---------|---------|
| 性能不达标 | 分析瓶颈（数据搬运/计算/流水线），调整 Tiling |
| 稀疏度导致性能退化 | 分析不同稀疏分布下的性能表现，优化行分配策略 |

CP3.2 为 ⛔ 必须询问用户，无论性能是否达标都需用户审批。不达标时启动 ops-profiling skill 排查：
1. 检查 MTE2/V/MTE3 利用率
2. 检查 Double Buffer 是否生效
3. 检查 bank conflict
4. 参考 ascendc-performance-best-practices skill 针对算子类型做专项优化

### 3.3 大 shape 精简失败

1. 检查精简后保留的用例参数是否合法
2. 尝试保留不同的大 shape 用例（如从 m=4096 改为 m=2048）
3. 若多次尝试仍失败，建议用户选择「保留全部」

## 阶段四：上库

### 4.1 编写文档

| 错误类型 | 处理方式 |
|---------|---------|
| 文档生成失败 | writer 重新生成，不询问用户 |

### 4.1.1 README 内容审查不通过

审查报告列出具体发现（带行号+期望值）。主 Agent 打回 writer 修复 README（不询问用户），然后重新运行 4.1.1。最多 2 次。

| 错误类型 | 处理方式 |
|---------|---------|
| 模板章节缺失 | writer 补充缺失章节后重新审查 |
| API 签名不匹配 | writer 对照 cann_ops_sparse.h 修正签名后重新审查 |
| 参数类型错误 | writer 修正参数类型后重新审查 |
| RAII 模式缺失 | writer 重写调用示例，使用 AclContext + unique_ptr 后重新审查 |
| API 名称错误 | writer 修正 API 名称后重新审查 |
| 头文件不完整/冗余 | writer 增删 #include 后重新审查 |
| 缺少 compile_and_run_example 链接 | writer 添加交叉引用链接后重新审查 |
| 内存位置注释错误 | writer 修正 Host/Device 标注后重新审查 |
| 约束描述不匹配 | writer 与 host.cpp 中 Validate 函数对齐约束后重新审查 |

### 4.1.2 README 编译测试不通过

编译测试失败时，主 Agent 打回 writer 修复 README 中的调用示例（不询问用户），然后重新运行 4.1.1 和 4.1.2。最多 2 次。NPU 不可用时标记为跳过，编译通过即视为成功。

| 错误类型 | 处理方式 |
|---------|---------|
| 编译错误（语法/类型） | writer 检查 include 和 C++ 语法后修复调用示例 |
| 链接错误（未定义符号） | writer 检查 API 名称匹配实际声明后修复 |
| CMake 配置错误 | writer 检查 CMakeLists.txt 模板适用性后修复 |
| NPU 不可用 | 跳过运行时，编译通过即可 |
| 运行时崩溃/结果错误 | writer 检查参数初始化、数据流和同步后修复调用示例 |

### 代码检视不通过（4.2）

修复问题后需重新执行 4.2 代码检视验证。
